### 技术指标要求

```
✅ 测量精度：
├── 长度/直径：误差 < ±0.1mm（对手机+标定板已经很好）
├── 角度：误差 < ±1°
└── 圆度：误差 < ±0.05

✅ 重复性（GR&R）：
├── 目标：GR&R < 10%（优秀级）
├── 底线：GR&R < 30%（可接受级）
└── 分析不达标的原因并写在报告里

✅ 工程指标：
├── 单张测量 < 500ms
├── API响应 < 1s（含图像传输）
├── Docker镜像 < 1GB
└── 测试覆盖率 > 80%

✅ 文档：
├── README 中英双语
├── API文档（Swagger自动生成）
├── 误差分析报告
└── Demo视频（3分钟）
```

### 面试话术模板

```
"我做过一个工业级视觉测量系统，对标Halcon的测量功能。

核心挑战不是算法——Canny+轮廓谁都会——而是测量系统的可靠性。

我引入了GR&R方法论评估系统：3个模拟操作员×10个零件×3次重复，
最终GR&R达到了8.2%（<10%优秀级）。

另外，我意识到算法只是系统的一部分。我用FastAPI把测量能力
封装成REST API，用Docker打包——这意味着换成任何工控机都能直接部署。

我还用Halcon对同一批零件做了对比测量，偏差在0.02mm以内，
这说明我的OpenCV方案在生产环境中是可用的替代方案。"
```

```

│   ├── EfficientNet-B3（轻量级对比）

│   ├── 表格：mAP/Recall/Precision/FPS/参数量/推理时间

│   │   面试价值：⭐⭐⭐⭐⭐（工业视觉前沿方向！）

├── Day 5-6：Focal Loss + 类别不平衡
│   ├── 对比实验：CE Loss vs Focal Loss vs Class Weight

│   └── 可视化：不同方法的混淆矩阵对比

📅 Week 14（🆕新增周）：产线模拟 + 集成 + Demo
├── Day 1-3：产线实时推理模拟

│   ├── 实时推理 + 缺陷标注 + 分类
│   ├── 模拟分拣信号：
│   │   ├── OK → 信号0（通过）
│   │   ├── Defect Type A → 信号1（分拣到A箱）
│   │   ├── Defect Type B → 信号2（分拣到B箱）
│   │   └── 用Modbus TCP模拟（pymodbus库）
│   ├── 实时统计面板：
│   │   ├── 总检测数 / OK数 / 缺陷数
│   │   ├── 缺陷类型分布（饼图）
│   │   ├── 过杀率 / 漏杀率（实时更新）
│   │   └── 平均推理时间
│   └── 代码产出：`production_line_simulator.py`
├── Day 4：Docker Compose一键部署
│   ├── 推理服务容器
│   ├── 模拟PLC容器（Modbus Server）
│   └── 一键启动：`docker-compose up`
├── Day 5：录制Demo视频 + 复盘文章

```
